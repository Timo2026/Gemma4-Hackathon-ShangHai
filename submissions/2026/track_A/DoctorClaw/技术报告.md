# DoctorClaw 技术报告

> **项目名称**：DocClaw / DoctorClaw 医疗 AI 工作台  
> **文档版本**：2026-06  
> **AI 引擎**：Google Gemma 4 开源模型家族  
> **核心主题**：Gemma 4 模型选型与系统架构设计

---

## 1. 执行摘要

DoctorClaw 是面向医生的医疗 AI 工作台，基于 **Google Gemma 4** 构建双轨智能体架构：Skill Runtime 负责高质量结构化病历生成，DeepAgents Harness 负责多轮工具编排与 HITL 人机协同。系统通过 DeepInfra / Google AI Studio 云端 OpenAI 兼容 API 接入 Gemma 4，并预留沐曦国产算力一体机本地 vLLM 部署能力。

**核心设计**：

- **31B Dense**（`google/gemma-4-31B-it`）— 病历结构化 Skill，json_schema 强制输出
- **26B MoE**（`google/gemma-4-26B-A4B-it`）— Agent 多轮工具调用，低延迟编排
- **统一多模态** — 问诊附件、ECG/影像解读走 Gemma 4 原生 vision 能力

## 2. 项目背景

### 2.1 产品定位

| 模块 | 能力 |
|------|------|
| 患者队列 | 今日问诊列表，状态/类型筛选 |
| 问诊工作台 | Skill + Agent 双模式；病历结构化、检查分析 |
| 技能管理 | 个人技能、技能广场、ClawHub 同步 |
| 随访计划 | APScheduler 自动/手动执行 |

### 2.2 为何选择 Gemma 4 作为底座

1. **开源可部署**：Apache 2.0 许可，支持云端 API 与本地 vLLM，契合沐曦国产算力终局；
2. **医疗场景匹配**：31B 规模满足结构化病历质量下限，MoE 变体满足 Agent 实时性；
3. **原生能力完备**：json_schema guided decoding、function calling、多模态（图文/音频）一体化；
4. **长上下文**：256K token，支撑长问诊对话与患者全周期档案。

## 3. Gemma 4 模型选型

### 3.1 型号对照

| 型号 | 参数量 | 激活参数 | 上下文 | DoctorClaw 用途 |
|------|--------|----------|--------|-----------------|
| Gemma 4 E2B / E4B | ~2–4B | 同左 | 128K | 未采用（临床质量不足） |
| Gemma 4 12B | 12B | 12B | — | 备选轻量部署 |
| **Gemma 4 26B-A4B IT** | 26B | **~3.8B** | 256K | **Agent Harness 主模型** |
| **Gemma 4 31B IT** | 31B | 31B | 256K | **病历结构化 Skill** |

### 3.2 病历 Skill：31B Dense

门诊病历结构化要求单次 LLM 调用输出 10 字段 JSON Schema，字段完整率 ≥ 95%，事实一致性严格。

**选型理由**：

- 31B Dense 全参数激活，推理能力对标医疗文档推荐的 32B 质量下限；
- `response_format: json_schema` + Pydantic 双重校验，JSON 解析成功率 ≥ 95%；
- 单次调用路径，可承受 31B 相对较高的推理时延；
- `enable_thinking: false`，避免 reasoning token 污染结构化正文。

### 3.3 Agent Harness：26B-A4B MoE

Agent 模式经主 Agent → 子 Agent → MCP 工具（18+）→ HITL，多轮请求累加。

**选型理由**：

- MoE 仅激活 ~3.8B 参数，多轮工具调用 P95 延迟显著低于 31B Dense；
- Gemma 4 原生 function calling，与 LangGraph / DeepAgents 工具链兼容；
- 主 Agent `temperature=1.0`，摘要中间件 `temperature=0.3`，符合官方采样建议。

### 3.4 部署：云端 API + 本地预留

当前默认 **DeepInfra OpenAI 兼容端点**；终局切换沐曦 METAX 本地 vLLM 时，仅需保持 `LLM_BASE_URL` 指向本地 `:8000/v1`，业务代码零改动。

## 4. 系统架构

### 4.1 四服务拓扑

| 服务 | 端口 | 职责 |
|------|------|------|
| Medical API | 8000 | 患者、问诊、技能、随访 |
| MCP Server | 8001 | Agent 医疗业务工具层 |
| Agent API | 8090 | SSE 流式 + HITL `/resume` |
| Frontend | 5173 | 医生工作台 UI |

### 4.2 双轨 AI 通路

**通路 A — Skill Runtime（质量优先）**

```
病历意图 → skill_runtime_stream → Gemma 4 31B（json_schema）
  → Pydantic 校验 + field_diff → SSE structured → 医生确认
```

**通路 B — DeepAgents Harness（编排优先）**

```
自然语言 → Agent API → Gemma 4 26B-A4B → 子 Agent → MCP 工具 → HITL
```

### 4.3 模型层

| 组件 | 路径 | 说明 |
|------|------|------|
| GemmaProvider | `services/llm/gemma_provider.py` | Skill 侧 HTTP 调用 |
| build_chat_model | `agent/config.py` | Agent 侧 LangChain 工厂 |
| FallbackLLMProvider | `services/llm/factory.py` | 无 Key 时 Mock 降级 |

**Gemma 适配要点**：

- `extra_body.chat_template_kwargs.enable_thinking = false`
- `reasoning` 字段剥离，仅输出 `content`
- `json_schema` 流式 buffer，完整 JSON 后一次性推送

### 4.4 多模态

`ChatMessage.content` 支持 OpenAI 多模态 parts；ConsultPage 图片附件以 base64 data URL 注入 Gemma 4 31B 请求，用于检查报告、ECG 影像辅助解读。

### 4.5 RAG 预留

`services/rag/` 定义 `RAGRetriever` 协议与 `StubRAGRetriever` 占位，赛后接入科室规范库 Top-K 注入，不改变双轨架构。

## 5. 质量保障

| 指标 | 目标 |
|------|------|
| 必填字段完整率 | ≥ 90% |
| 事实性错误率 | ≤ 5% |
| JSON 解析成功率 | ≥ 95% |
| 端到端延迟 P95 | ≤ 3s（云端 API） |

护栏链：json_schema 约束 → Pydantic 校验 → field_diff 事实比对 → 医生 HITL 必审。

## 6. 配置参考

```env
LLM_PROVIDER=gemma
LLM_MODEL=google/gemma-4-31B-it
LLM_BASE_URL=https://api.deepinfra.com/v1/openai
LLM_STRUCTURED_OUTPUT_MODE=json_schema

AGENT_MODEL=google/gemma-4-26B-A4B-it
AGENT_BASE_URL=https://api.deepinfra.com/v1/openai
```

验收：`python scripts/e2e_acceptance.py --with-llm --with-agent`

## 7. 结论

DoctorClaw 从架构设计之初即以 **Gemma 4 双型号分场景路由** 为核心：31B 保障病历结构化质量，26B MoE 保障 Agent 实时编排。OpenAI 兼容 API 层屏蔽部署差异，为云端演示与沐曦本地推理提供统一接入面。
