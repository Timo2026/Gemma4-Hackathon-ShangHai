# ViewWander · 技术报告

> Gemma 4 开发者大赛 2026 · B 赛道 · 多模态
> 把《极乐迪斯科》的内心独白，塞进相机的取景器。

---

## 1. 项目概述

ViewWander 是一个相机应用。按一次快门拍下取景器里的一帧画面，Gemma 4 会扮演一组住在脑子里
的人格，对这帧画面接力吐出内心独白——如同《极乐迪斯科》中逻辑、本能、共情、超现实直觉轮番
插话。

它针对的不是“这是什么”的识别问题，而是“这一帧让人联想到什么”的表达问题。对同一张街景，
逻辑关注几何与因果，本能关注威胁与生存，超现实人格关注诗意与神秘；一张图由此并置出多种内在
声音。与之相对，常见相机应用的视觉理解止于物体标注，ViewWander 把多模态理解推进到主观解读与
情绪投射这一层。

一项贯穿全局的设计约束是：模型始终接收完整的 3:4 帧，而用户只看到居中的 1:1 方格。人格所
解读的内容刻意包含取景框之外的部分——“画外之意”是本项目的核心设计意图，也对模型的空间理解
构成真实考验。

### 1.1 本提交包

完整产品是一个浏览器相机应用，包含 React 前端、端侧与云端双后端及 Vercel 部署。为便于评审以
单条命令复现多模态效果，本提交将核心推理逻辑抽取为一组自包含、零第三方依赖的命令行 Python
程序，其文件组织如表 1。

**表 1 · 提交代码的文件组织**

| 文件 | 角色 |
|---|---|
| [`main.py`](../main.py) | 交互入口。运行时选择云端或本地后端，云端模式索取 API key，随后分发至对应后端 |
| [`src/common.py`](../src/common.py) | 共享核心：人格名册解析、导演 prompt 构造、两段式接力编排、选人结果定型 |
| [`src/cloud.py`](../src/cloud.py) | 云端后端：Google AI Studio 原生 Gemini API，模型 `gemma-4-26b-a4b-it` |
| [`src/local.py`](../src/local.py) | 端侧后端：本机 oMLX，OpenAI 兼容接口，模型 `gemma-4-e4b-it-4bit` |
| [`src/data/`](../src/data/) | 14 张测试图与 `labels.jsonl` 人工标注，满足 B 赛道可复现要求 |

系统的选型与实现，凡涉及具体代码处均标注其在 `src/` 中的位置。

## 2. 模型选型

ViewWander 同时落地两档 Gemma 4，分别服务两类场景。这并非冗余，而是对“端侧响应手感”与“云端
可访问性”两个目标的分工，如表 2。

**表 2 · 两档模型的选型**

| 场景 | 规格 | 后端 | 选型理由 |
|---|---|---|---|
| 端侧 | `gemma-4-e4b-it-4bit` | 本机 oMLX（Mac 原生 MLX server） | 4.5B 有效参数，4bit 量化，约 5GB 占用，在 24GB 统一内存的 Mac 上运行宽裕。按快门后秒级出独白的响应手感依赖该档的低首 token 时延；照片不离开设备，断网亦可运行 |
| 云端 | `gemma-4-26b-a4b-it` | Google AI Studio 托管 | 26B MoE 架构，激活约 4B 参数。评委与移动端用户无需本机环境即可访问，本机可完全关机。AI Studio 不提供 e4b，在其可选的大尺寸中主动选 26B MoE：仅激活约 4B 参数，在质量与生成时延间取得平衡（详见下文），也使演示得以展示 Gemma 4 MoE 档的视觉理解质量 |

不采用 31B Dense、也不在本机运行 26B 的原因如下。其一，本机内存处于边缘：26B 量化后逼近 24GB
上限，与浏览器争用内存且首 token 时延显著上升，破坏按快门即出的实时手感；而 e4b 的绝对质量
损失在“主观联想”这一无标准答案的任务中可以接受，低时延带来的体验收益更大。其二，在 AI Studio
上 26B 与 31B 同为免费、价格无别，云端选型因此不取决于成本，而取决于生成时延：31B Dense 每
token 激活全部参数，26B MoE 仅激活约 4B 参数，在本系统这条单次快门触发多次调用的流式接力中，
MoE 的逐 token 生成更快；而 Dense 在该主观任务上的质量增益不足以抵消其更低的吞吐。

综上，端侧以足够轻量的 e4b 换取即时手感，云端以 AI Studio 唯一可选的 26B MoE 换取零门槛
可访问；两档共用同一套编排核心，切换成本被压缩为一个开关。

## 3. 系统架构

本系统采用 A 路线，即编排式架构：在前端产品中表现为前端编排，在本提交中表现为应用层编排。

### 3.1 编排与推理的分离

核心设计是将编排与单步推理解耦。编排——即决定调用哪些人格、以何种顺序调用、如何在它们之间
传递上文——由应用层的确定性代码完成，而不交由模型的 agentic 多步循环；模型内部的 MoE expert
路由属不可寻址的黑盒，亦不作为编排手段。模型只承担两类纯认知任务：人格选择与独白生成。

需要厘清的是，“编排不交给模型”与“某一步使用函数调用”并不矛盾：前者指多步流程的控制权归于
应用层，后者指人格选择这一**单步**推理以 Gemma 4 原生函数调用产出结构化结果（见 §5.1）。二者
位于不同层次。

整条推理链路如图 1，由 `common.run_relay()` 编排：后端 `pick()` 选人，`finalize_pick()` 定型，
再逐个调用后端 `voice()` 生成独白，与前端产品同构。

**图 1 · 单次快门的推理链路**

```
[拍照 · 完整 3:4 帧 base64]
        │
        ▼
① pick(image)  ──→  { personas: id[2-4], context }
   导演看图，从 17 人格池中挑出被该帧强触发的 2–4 个，并写一句场景纸条
        │
        ▼
finalize_pick：校验 id → 按名册 order 升序重排 → 截断至至多 4 个
   固定的全序韵律为：本能/生理 → 智力 → 社会 → 超现实，volition 居末以收束
        │
        ▼
② for each persona in lineup:
     voice(persona, context, priorVoices, image)
       该人格基于图像、场景与前文，生成 1–2 句独白
```

### 3.2 两段式接力推理

单次快门由 1 次导演选人（`pick`）与 N 次人格独白（`voice`，N 为 2–4）串联而成。

- 导演阶段：模型看图并调用 `cast_personas(personas, context)` 工具提交选择（原生函数调用，
  见 §5.1）。导演 prompt 由 `build_pick_prompt()` 遍历 17 人格名册、动态拼入各人格的 `trigger`
  字段，因此新增人格无需改动导演代码。
- 人格阶段：每个人格基于图像、场景以及在它之前已发言的人格的内容，生成 1–2 句独白。

### 3.3 接力上下文的构造

在后续人格的 user message 中，先前人格已发言的内容以 `[前文已有]` 段作为前缀注入（见
`common.py:build_user_message`）。接力感由这一编排步骤构造，而非模型自动续写——每个人格的声口
由各自的 system prompt 独立掌控，同时能够“听见”前文，从而形成对话感。

### 3.4 数据驱动的人格名册

人格的事实源是 [`src/personas/*.md`](../src/personas/)，一个人格对应一个 Markdown 文件（frontmatter
记录元信息，正文为声口 prompt）。`load_personas()` 在启动时解析这些文件：`build_pick_prompt()`
将每个人格的 `trigger` 拼入导演 prompt，正文则作为该人格生成独白时的 system prompt。新增人格
即新增一个文件，无需改动编排逻辑；项目据此从 4 人格平滑扩展至 17 人格，编排代码未作改动。
（前端产品另设 codegen 步骤将 Markdown 编译为纯数据，与本脚本运行时直接解析等价。）

### 3.5 共享核心与双后端

这是本提交的结构主线：人格名册解析、导演 prompt 构造、接力编排与选人定型全部置于 `common.py`；
云端与端侧仅在最底层一次 HTTP 请求上不同，分别收于 `cloud.py` 与 `local.py`。两条后端只需实现
一个极薄的契约（duck typing，无需继承）：`encode_image`、`pick`、`voice` 与 `DEFAULT_MODEL`。
两者的差异完整列于表 3。

**表 3 · 云端与端侧后端的差异**

| 维度 | 云端（`cloud.py`） | 端侧（`local.py`） |
|---|---|---|
| 服务地址 | `generativelanguage.googleapis.com` | `localhost:8080`（本机 oMLX） |
| 接口格式 | 原生 Gemini（`systemInstruction` + `inlineData`） | OpenAI 兼容（`messages` + `image_url`） |
| 模型 | `gemma-4-26b-a4b-it`（26B MoE） | `gemma-4-e4b-it-4bit`（E4B） |
| 图像载体 | `inlineData {mimeType, data}`，图像置于文本之前 | `image_url`，base64 data URL |
| pick 函数调用 | `functionDeclarations` + `toolConfig: ANY` | OpenAI `tools` + `tool_choice` |
| 强制思考 | 强制开启，遍历 parts 跳过 `thought:true`（§5.2） | 不强制思考，`content` 即正文 |
| 鉴权 | `x-goog-api-key`（环境变量或交互索取） | `Bearer local`（本地占位，不联网） |
| pick 实测时延 | 约 110s（思考开销大，timeout 设为 200s） | 约 3s |

将编排与单次 HTTP 请求彻底分离，是这套代码能够同时落地云端 26B MoE 与端侧 E4B 两档的关键：
新增一档后端等价于新增一个实现上述四个方法的文件，编排核心无需改动。

## 4. 多模态处理

B 赛道的核心在于多模态处理，本系统在以下三点上作了具体处理。

第一，采用真实视觉输入，而非先生成图像描述再处理的二段式方案。每次推理都将 3:4 帧的 base64
直接送入模型——导演看图选人，每个人格亦各自看图生成独白；视觉与文本在同一次前向中融合，而非
先将图像转写为文字。图像载体随后端而异（见表 3）：云端用 `inlineData`，端侧用 `image_url`。

第二，模型输入帧与用户可见帧不一致（3:4 与 1:1）。多模态理解被刻意用于解读用户未主动框入的
部分，这既是产品的核心意图，也是对模型空间理解的真实压力测试。

第三，端侧的图像编码缓存经过实测。在单次快门的 N+1 次请求中，本机 oMLX 按图像内容寻址缓存：
仅导演那次为冷编码（约 1.4s），其余人格命中缓存（约 0.7s）。该结论支持了“不引入额外的前缀
对齐优化”这一决定（更换后端需重新测定）。

## 5. 对 Gemma 4 原生能力的运用

本节说明系统如何运用 Gemma 4 的原生能力，而非停留于 prompt 工程。

### 5.1 原生函数调用与结构化人格选择

导演选人不采用“自由文本加正则或 JSON 解析”的方式，而是为 Gemma 4 提供 `cast_personas(personas,
context)` 工具，强制其经函数调用通道提交选择。其要点为：

- `personas` 字段以 enum 锁定 17 个合法 id，约束解码（constrained decoding）从根本上杜绝模型
  虚构人格；
- 直接获取结构化的调用参数，免去 JSON 解析，并因此天然规避 markdown 代码围栏导致的解析失败；
- 两条后端各自使用其原生协议（见表 3）：端侧为 OpenAI `tools`/`tool_choice`，云端为原生
  `functionDeclarations`/`toolConfig: ANY`；二者的选人逻辑相同（各自的 `build_cast_tool`，以及
  云端 `_call_gemma_pick` 与端侧 `_call_local_pick`）。

实现中有两处需要注意。其一，prompt 不应与工具争夺输出通道：实测表明，若导演 prompt 仍含“返回
JSON”一类指令，模型会转而输出文本 JSON 而不走函数调用，故 `build_pick_prompt(output_mode='tool')`
移除了 JSON 相关表述，选人逻辑不变，仅更换输出契约。其二，结果位于 `tool_calls` 而非 `content`：
oMLX 在工具模式下会将 `<eos>` 泄漏入 `content`，因此必须从 `tool_calls[].function.arguments`
读取参数，不可读取 `content`。

两条后端的提交脚本均以函数调用执行 pick，原因在于时延权衡。云端 26B 强制思考（见 §5.2），
函数调用模式下 pick 时延上升至约 110s，而端侧 e4b 仅约 3s。完整前端产品据此采取端侧用函数
调用、云端前端保留自由文本（以保证交互流畅）的策略；本提交脚本则两条后端的 pick 均用函数
调用——云端离线复现允许较慢，以换取评审一次即可观察到原生函数调用，端侧本就足够快。两种契约
下的选人质量基本持平，量化对照见 §6。

### 5.2 云端强制思考的处理

云端 Gemma 4（26B 与 31B）的思考为强制开启且无法关闭——三类 API 开关均经测试，均不能完全
关闭。处理该行为是云端路径的关键，做法有二。

其一，采用原生 Gemini API 而非 OpenAI 兼容层。OpenAI 兼容层将思考与正文合并于同一 `content`
字段，只能以正则强行剥离；原生 API 则将思考置于独立的 `thought: true` part，遍历响应时跳过
该 part 即可干净保留正文（见 `cloud.py:_call_gemma` 中的 `if not p.get("thought")` 判断）。
这是让强制思考的模型仍能输出干净独白的工程关键，而非依赖 prompt 要求模型“不要输出思考”。

其二，`maxOutputTokens` 需给足（pick 为 4096，voice 为 1024）。思考会先消耗数百至上千 token
才轮到正文，配额过小将导致正文无法输出（`finishReason: MAX_TOKENS`）。

端侧 e4b 不强制思考，`local.py:_call_local` 直接读取 `content`，无此开销——该处理为云端路径
所独有，亦即表 3 中“强制思考”一行两侧差异的来源。`src/data/thinking-sample.json` 保存了一份
真实的原生思考响应，可直观查看 `thought:true` part 的结构。

补充说明：完整前端产品中，云端路径另有一层 Vercel Edge 代理（在服务端注入 key、改写 `/v1` 为
`/v1beta`、透传 SSE，使移动端可同源访问且 key 不进入打包产物）。提交脚本省去该代理，直接调用
原生 API，处理强制思考的核心逻辑与前端完全一致。

## 6. 评测与工程质量

系统的工程质量要点列于表 4。

**表 4 · 工程质量要点**

| 要点 | 说明 |
|---|---|
| 选人质量评测闭环 | 以“平均选人数、每人格出场直方图、required 命中率、forbid 干净率、JSON 解析成功率”量化导演选人，在调整 prompt 或更换调用方式前先建立基线（标注源为 `src/data/labels.jsonl`），对照见下文 |
| 可复现的测试素材 | 14 张测试图与三档人工标注（required、forbid、acceptable），评委可据此复现并评估选人质量，满足 B 赛道对测试素材的要求 |
| 共享核心与双后端 | 编排逻辑全部置于 `common.py`，云端与端侧仅差一次 HTTP（见 §3.5），双后端是可运行的两条路径而非冗余拷贝 |
| 零第三方依赖 | `src/` 仅使用 Python 标准库（urllib、base64、json），将 Gemma 4 调用逻辑完整暴露，不封装于 SDK 之后 |
| 单一事实源 | 导演 prompt 由 `build_pick_prompt()` 从人格名册动态拼装，增改人格仅需修改 Markdown，无副本漂移 |

将 pick 切换为函数调用的决策，建立在两类证据之上：输出健壮性，以及受控条件下的选人质量对照。

其一为输出健壮性。在默认采样温度下，自由文本方案的裸 JSON 解析成功率约为 71%，失败主要源于
markdown 代码围栏污染；函数调用由结构化输出保证，解析成功率为 100%。这一结构化保证是采用函数
调用的主要动机。

其二为受控条件下的选人质量对照，结果见表 5。为消除小模型在默认温度下的采样噪声，对照在端侧
e4b、温度为 0、全图集 14 张图的条件下进行（此条件下两种方案的裸解析均为 14/14）。

**表 5 · 函数调用与自由文本的选人质量对照（端侧 e4b，温度 0，14 图）**

| 指标 | 自由文本 | 函数调用 |
|---|---|---|
| 整体 pass | 50.0% | 50.0% |
| forbid 干净率（越高越好） | 57.1% | 71.4% |
| required 命中率 | 64.3% | 57.1% |
| inland 霸屏次数 | 14/14 | 12/14 |
| 裸 JSON 解析成功 | 14/14 | 14/14 |

解读该表前需先厘清这把尺子的边界。标注由开发者本人完成，为单标注者、无标注者间一致性
（inter-annotator agreement）核验；联想本身是主观任务，required 与 forbid 仅标注了少数明显
该触发或明显不该触发的人格，acceptable 灰区刻意留宽。因此该对照衡量的是“是否踩中明显的对与
错”，而非选人质量的绝对标尺。加之样本仅 14 张（单张图约合 7 个百分点）、温度 0 仅单次运行、
无法获得运行间方差，下列数字应作描述性观察，不足以支撑显著性结论。

在上述条件下观察到：整体 pass 两侧均为 7/14；forbid 干净由 8/14 升至 10/14（移动 2 张图）；
inland 出场由 14/14 降至 12/14（移动 2 张图）；裸 JSON 解析两侧均为 14/14。代价一侧亦如实
列出：required 命中由 9/14 降至 8/14（反向移动 1 张图）。即函数调用在“减少 forbid 越界”上
移动了 1–2 张图，同时在“凑齐 required”上让出 1 张图，是一笔可见的取舍而非全面改善，且均处于
单张图约 7 个百分点的噪声量级内——方向暗示大于统计证据。综合健壮性收益与该对照，系统选择函数
调用作为 pick 的输出契约：其选人质量在这把粗粒度尺子上未见可见退化。

## 7. 复现说明

提交代码零依赖、无需环境配置即可运行，云端模式仅需一把 API key：

```bash
pip install -r requirements.txt   # 零依赖，直接通过
python main.py                    # 交互：选云端或本地，云端粘贴 key，选图后运行
```

`main.py` 交互选择云端即评审的最快路径（真实调用 26B MoE，无需本地环境）。亦可跳过交互、直接
运行某条后端：

```bash
export GEMMA_API_KEY=<AI Studio key>   # https://aistudio.google.com/apikey
python src/cloud.py                  # 云端 26B，默认分析 data/1.jpg
python src/cloud.py --all            # 云端 26B，遍历全部测试图
python src/local.py --all            # 端侧 e4b，需本机 oMLX 常驻 8080，照片不离设备
```

运行要求 Python 3.8 及以上。完整运行示例见 [`README.md`](../README.md)。

完整前端产品（本提交未含其源码）为浏览器相机应用：`mock` 模式零后端即可运行，`mlx` 模式接入
本机 oMLX（端侧 e4b），`cloud` 模式经 Vercel 与 AI Studio（云端 26B，为移动端与评委访问的
主路径）。在线体验地址见 README。

## 8. 隐私与合规

- 云端模式下，照片与 prompt 会传输至 Google AI Studio；免费层数据可能用于训练且无可关闭的付费
  档位。这是演示阶段的已知取舍，正式产品应切换至自托管或付费档位。
- 端侧模式下，推理全程在本机 oMLX 完成，照片不离开设备、断网可运行，适用于隐私敏感场景。保留
  `local.py` 这一可运行路径正是为此。

## 9. 技术栈

- 核心复现代码：Python 3.8+ 标准库（`main.py` 交互入口、`src/` 共享核心与双后端），零第三方依赖。
- 端侧推理：本机 oMLX，OpenAI 兼容接口，SSE 流式，模型 `gemma-4-e4b-it-4bit`。
- 云端推理：Google AI Studio 原生 Gemini API，模型 `gemma-4-26b-a4b-it`；前端产品另经 Vercel Edge 代理。
- 完整前端：React 19、Vite 8、framer-motion、TypeScript。
- 架构：A 路线，编排式（应用层编排，不使用 MoE expert 路由）；pick 选人采用原生函数调用。
